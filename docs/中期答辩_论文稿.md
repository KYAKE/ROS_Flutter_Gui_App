## 一、课题研究背景与意义

随着自动驾驶车辆、移动机器人及智能装备系统的发展，系统运行状态的可视化、任务执行过程的交互化以及控制界面的跨平台部署能力，逐渐成为机器人系统工程化应用中的重要环节。当前，机器人系统开发与调试通常以 ROS 生态为核心，常见的交互手段包括命令行工具、RViz、rqt 以及若干针对单一平台开发的专用界面。这些工具在专业研发人员使用场景下具有较强功能性，但在自动驾驶测试、移动机器人管理、跨设备监控和非专业用户操作等实际应用场景中，仍然存在以下不足：一是使用门槛较高，系统操作分散于多个窗口与工具之间；二是交互流程不统一，难以形成面向任务执行的完整操作闭环；三是跨平台能力不足，同一系统往往需要分别开发桌面端、Web 端或移动端界面；四是运维监控、可视化展示与远程控制能力耦合度低，不利于系统快速部署与展示。

基于上述问题，本课题围绕“跨平台”“可视化”“可交互”“面向 ROS 系统”四个关键词展开研究，拟设计并实现一个基于 Flutter 与 ROS 的自动驾驶跨平台人机交互 App。该系统以 Flutter 作为前端跨平台开发框架，以 ROS/ROS2 作为机器人系统运行环境，以 rosbridge WebSocket 作为通信桥梁，将地图显示、机器人状态监测、导航任务下发、手动控制、诊断展示、相机视频流查看以及拓扑地图编辑等功能集成到统一界面中，从而形成一个兼顾工程实用性与研究探索价值的机器人 HMI 原型系统。

从理论意义来看，本课题有助于探索 Flutter 在机器人人机交互界面领域中的适用性，并进一步分析其在 ROS 场景下的跨平台开发优势、实时可视化能力和工程部署潜力。从工程意义来看，若能够形成较完整的软件原型，则可以为自动驾驶测试车辆、室内服务机器人、巡检机器人等场景提供一套更轻量、更直观、更易部署的人机交互解决方案。尤其是在需要快速搭建演示系统、构建多平台控制终端或面向非开发人员提供操作界面的场合，该类系统具备较高的现实应用价值。

因此，本课题不仅具有明确的软件工程实现目标，同时也具备较强的研究和应用导向，其核心任务是在 ROS 生态与现代跨平台前端框架之间建立高效、稳定、可扩展的人机交互通道，并在此基础上形成较完整的功能验证与论文研究成果。

## 三、当前系统总体设计与关键技术实现

### 3.1 系统总体架构

当前系统采用“Flutter 前端 + ROS/ROS2 后端 + rosbridge WebSocket 通信 + Flame 2D 场景渲染”的总体架构。该架构的出发点在于：前端侧需要支持 Android、Web、Linux、Windows 等多平台统一开发，后端侧则尽量保持 ROS 系统原有节点结构不变，通过标准主题、服务和坐标变换消息与前端交互。因此，系统没有采用直接嵌入 ROS 原生客户端库的方式，而是利用 rosbridge 对 ROS 消息进行桥接，使 Flutter 端以 WebSocket 客户端的形式完成数据订阅与控制指令发布。

在这一架构中，Flutter 负责页面组织、用户交互、状态管理和跨平台运行；Flame 负责地图场景、路径、激光点、点云点集、机器人位姿等内容的 2D 绘制与场景更新；ROS 侧负责提供实时运行数据和执行控制指令；`roslibdart` 则作为前端侧的 ROS 通信基础能力封装。该方案的优势在于代码结构清晰、系统解耦程度较高、平台迁移成本较低，且能够较好地适配机器人 HMI 场景中“实时可视化 + 交互控制 + 多平台部署”的综合需求。

### 3.2 通信模块设计与实现

通信模块是本课题的核心部分之一。当前项目已围绕 ROS 通信过程构建了较为完整的封装逻辑，其中 `RosChannel` 模块承担了通信调度中心角色，负责 ROS 连接建立、连接状态维护、消息订阅、消息发布、连接异常处理以及业务数据向界面层的分发。为保证系统在实际使用中的可用性，当前代码中已经实现了连接状态切换和断线重连逻辑，当网络波动或 ROS 侧连接中断时，系统会尝试自动重连，并通过界面反馈连接状态变化，从而提高联调和运行时的稳定性。

就消息类型支持而言，当前系统已能够处理多类与自动驾驶和移动机器人应用密切相关的标准 ROS 消息。具体包括：用于地图显示的 `nav_msgs/OccupancyGrid`，用于局部路径、全局路径和轨迹显示的 `nav_msgs/Path`，用于坐标变换的 `tf2_msgs/TFMessage`，用于激光雷达显示的 `sensor_msgs/LaserScan`，用于点云显示的 `sensor_msgs/PointCloud2`，用于电池状态监控的 `sensor_msgs/BatteryState`，用于姿态与运动信息获取的 `nav_msgs/Odometry`，用于机器人轮廓显示的 `geometry_msgs/PolygonStamped`，以及用于设备健康状态监测的 `diagnostic_msgs/DiagnosticArray` 等。此外，系统还可向 ROS 侧发布 `geometry_msgs/Twist` 速度控制消息、导航目标消息、重定位消息、地图更新消息和拓扑地图更新消息，实现真正意义上的双向交互。

这一阶段的实现表明，Flutter 端不仅能够作为被动数据显示界面存在，而且已经能够承担 ROS 系统中“监测—分析—操作—反馈”的前端交互角色。这是本课题从普通界面开发走向机器人应用系统开发的关键转折点。

### 3.3 地图与场景可视化设计

地图可视化是本课题区别于普通信息管理类 App 的重点内容，也是系统工程价值的直接体现。当前项目已基于 Flame 构建实时 2D 场景渲染能力，在统一地图视图中集成显示全局栅格地图、局部路径、全局路径、历史轨迹、激光雷达、点云数据、机器人姿态、机器人轮廓、局部代价地图、全局代价地图、导航点以及拓扑结构等内容。相比传统将这些信息分散显示在 RViz 不同图层或多个工具中的方式，本系统强调面向任务的统一展示，使操作人员能够在一个界面内同时观察机器人环境、当前位置、导航轨迹和感知状态。

为了提高界面的灵活性和可读性，系统进一步实现了图层控制逻辑，允许用户根据需要选择显示或隐藏路径、轨迹、激光、点云、拓扑图等图层。这一设计既有助于降低复杂信息叠加后的视觉混乱，也便于在不同测试任务中快速切换观察重点。通过当前阶段的实现可以看出，Flutter 虽然最初主要面向通用跨平台应用开发，但在结合 Flame 后，也能够承担较复杂的机器人 2D 场景绘制任务，这为后续论文中分析其在机器人 HMI 场景中的适用性提供了重要依据。

### 3.4 人机交互界面与任务控制设计

在界面结构上，当前系统已形成较清晰的页面分工。连接页主要用于配置 ROS Bridge 地址并建立连接，作为进入系统的入口；主控页承担地图显示、机器人状态显示和控制操作，是系统最核心的工作界面；设置页用于管理主题模式、语言、话题名称、视频流端口及其他配置项；诊断页用于集中展示 `/diagnostics` 主题信息，并进行分类汇总；地图编辑页则负责导航点编辑、拓扑结构编辑与保存。整体页面流程已经从单纯的原型展示逐步发展为较完整的应用工作流。

在交互控制层面，系统已实现虚拟摇杆控制、导航目标下发、导航取消、机器人重定位和急停控制等关键功能。其中，虚拟摇杆通过发布 `geometry_msgs/Twist` 指令实现机器人线速度与角速度控制，适合在调试和近距离遥控场景下使用；导航目标下发功能则更适用于自动驾驶或移动机器人自主运行任务；重定位功能可以在定位漂移或初始化场景下进行人工修正；急停功能则为系统提供基础安全控制手段。这些功能的实现使系统完成了从“只读式监控”向“可交互控制平台”的转变。

### 3.5 状态监测、诊断与辅助功能

为了提高系统在真实场景中的实用性，当前项目还实现了多种状态监测和辅助功能。其一，系统可实时显示电池状态、机器人速度、导航状态等运行信息，使用户能够快速掌握机器人当前工作状态；其二，系统能够解析并展示 `/diagnostics` 主题，将错误、警告、正常和失活状态进行分类汇总，并结合界面提示和 Toast 弹窗对异常情况进行反馈；其三，系统支持通过 `web_video_server` 查看摄像头视频流，弥补单纯地图显示在感知直观性方面的不足；其四，系统支持主题切换、多语言切换以及配置项持久化保存，这些功能虽然不直接改变 ROS 交互逻辑，但对软件工程完整性和使用体验提升具有明显作用。

综上，当前系统已经不仅是“ROS 数据展示面板”，而是一个集连接管理、场景可视化、控制交互、状态监测与参数配置于一体的跨平台机器人 HMI 原型系统。

## 四、当前开发环境与联调基础

结合当前项目进展，课题开发环境已逐步明确并向实际联调场景靠拢。现阶段主要以 `Ubuntu 24.04` 作为本地开发环境，前端采用 `Flutter 3.27.4`，并启用 Linux Desktop 作为桌面端验证平台。由于该项目需要在 Linux 桌面环境下运行，因此已围绕 `libgtk-3-dev`、`pkg-config`、`clang`、`cmake`、`ninja-build` 等依赖完成桌面构建环境梳理。与此同时，针对项目在依赖解析和构建过程中暴露出的 `Dart SDK` 约束、`GTK` 缺失、`CMake` 安装前缀缓存异常等问题，已经完成初步定位和修复方案整理，这使得后续本地迭代开发和中期后的实验环境搭建更加清晰可控。

在 ROS 侧，当前联调环境以 `ROS 2 Jazzy` 为主要目标版本，并围绕 `rosbridge_suite`、`web_video_server`、`ros-gz` 等组件开展环境准备。其中，`rosbridge_suite` 是本项目能够与 ROS 系统建立通信的核心依赖，`web_video_server` 用于相机图像流接入，`ros-gz` 和 Gazebo 相关组件则用于后续仿真场景联调。就系统架构而言，本课题项目通过 WebSocket 接入 ROS，因此前端对底层 ROS 节点实现并无强耦合，只要后端能够正确发布地图、路径、里程计、TF、诊断等标准消息，并启动 rosbridge 服务，前端即可接入并完成显示与控制。这一特点显著增强了系统的通用性和可移植性，也降低了与不同机器人平台联调时的适配成本。

从实验条件准备角度看，当前已基本具备后续开展仿真测试和实机联调的技术条件。下一阶段可在 RViz、Gazebo、Nav2 仿真环境以及 rosbridge 服务基础上，对地图显示、导航任务下发、路径更新、视频流接入和诊断监测等功能进行完整验证，并记录测试过程作为论文实验部分的直接支撑材料。

## 六、当前存在的问题与原因分析

尽管系统主体已基本完成，但中期后仍存在若干需要继续解决的问题，这些问题主要集中在性能、兼容性、联调复杂度和实验支撑深度四个方面。

首先，在性能方面，大尺寸地图、高频路径刷新、连续激光数据与点云数据叠加显示时，前端渲染负载明显增加。当前系统虽然已经能够实现相关显示功能，但在数据量进一步增大时仍可能出现局部卡顿、刷新延迟或界面交互流畅性下降的问题。这说明系统在数据节流、增量更新、渲染策略优化等方面仍有进一步提升空间。该问题的存在并不否定当前技术方案的可行性，但要求后续在论文中对其局限性做出客观分析，并通过优化实验展示改进效果。

其次，在通信方面，rosbridge 方案虽然部署灵活、接入简单，但其在复杂消息场景、高频数据流场景下的性能和稳定性仍存在一定约束。尤其当消息体积较大或消息频率较高时，前端解析成本增加，可能对交互实时性造成影响。此外，项目中也已识别到 `cbor` 数据、地图压缩以及部分 ROS1/ROS2 兼容细节问题，这意味着后续仍需继续完善消息处理鲁棒性。换言之，当前通信方案是“工程上可行且适配跨平台前端”的方案，但并非不存在瓶颈，这一点需要在后续论文分析中予以充分说明。

再次，在多平台适配方面，Flutter 的“一套代码多端运行”优势已经体现出来，但不同平台在输入行为、窗口布局、媒体组件支持和系统库依赖方面仍存在差异。例如，Linux 桌面构建依赖 GTK 相关环境，视频流显示在 Web 端和桌面端的支持程度存在差异，手柄输入能力也可能受平台插件支持影响。这意味着系统虽然已具备跨平台基础，但要实现高质量一致体验仍需要进一步打磨。

最后，在论文层面，目前最大的不足并非代码实现，而是实验材料和分析深度尚未完全跟上工程进展。当前已有较完整的软件原型和页面实现，但如果要形成一篇质量较高的毕业论文，还需要围绕“为什么采用 Flutter”“该系统相较传统 ROS 工具的优势体现在哪里”“系统在真实或仿真场景中表现如何”“存在哪些限制与改进方向”等问题，补充更系统的实验记录和分析论证。只有这样，才能使毕业论文从“工程实现说明”提升为“具备研究深度的系统设计论文”。

## 七、后期研究计划与时间安排

接下来的工作将重点围绕“系统优化—联调测试—实验总结—论文定稿”四个阶段展开。

第一阶段为系统优化与问题收敛阶段，计划在 2026 年 3 月中下旬完成。该阶段主要工作包括：继续优化大地图和多图层叠加场景下的渲染效率；完善 ROS2 Jazzy 环境下的消息兼容和本地联调脚本；进一步整理设置项、图层开关、页面交互细节，使系统在连续使用时更加稳定；对已知依赖问题和构建问题进行统一归档，降低后续环境重建成本。此阶段的目标是尽快冻结主体功能，避免后期继续进行无计划功能扩展。

第二阶段为仿真联调与实验记录阶段，计划在 2026 年 4 月上旬至中旬完成。该阶段拟基于 `rosbridge_server`、`web_video_server`、`RViz2`、`Gazebo Harmonic` 以及 Nav2 或相关仿真场景，开展面向功能完整性的系统验证。重点测试内容包括：连接建立与断线重连是否稳定、地图和路径显示是否正确、导航目标下发是否有效、速度控制是否能够正确作用于机器人、诊断信息是否能够及时分类显示、视频流是否能正常接入，以及不同平台或不同窗口尺寸下的界面适配情况。测试结果将通过截图、功能记录表和问题分析表的形式整理，为论文实验章节提供支撑。

第三阶段为论文主体撰写阶段，计划在 2026 年 4 月下旬至 5 月上旬完成。论文将围绕研究背景与意义、国内外研究现状、系统需求分析、总体架构设计、关键模块实现、实验结果与分析、总结与展望等内容展开。写作过程中将尽量做到“代码实现—实验现象—问题分析—结论提炼”相互对应，避免论文仅停留在功能罗列层面。

第四阶段为修改完善与答辩准备阶段，计划在 2026 年 5 月中旬完成。该阶段将在指导教师意见基础上修改论文，补充图表与运行截图，完善语言表达和格式规范；同时准备中后期答辩所需的系统演示环境、讲稿和展示材料，确保课题成果能够从论文文本和软件演示两个层面完整呈现。

一方面，最关键的不确定性已经基本消除。开题阶段的核心疑问在于：Flutter 是否适合承担机器人 HMI 前端任务，ROS 与 Flutter 之间能否建立稳定可用的通信链路，多图层地图显示与控制交互是否能够在统一应用中实现。从当前项目进展来看，这三个关键问题均已得到肯定验证，说明课题总体技术路线是成立的。既然系统主干已经形成，后续工作重点就转向“优化、测试、总结”，而不再是重新选型或重构系统，这使得项目整体收尾风险明显下降。

当然，也应当看到潜在风险主要集中在两个方面：其一，若联调阶段出现新的兼容性问题，可能会占用部分论文撰写时间；其二，若后期继续过度扩展功能，可能分散精力，影响收尾效率。因此，后续必须坚持“控制功能边界、优先完善实验、同步推进论文”的策略，将重点放在已完成功能的稳定化和论文化上，而不是持续增加新的展示性模块。

总体而言，本课题目前已经具备较好的完成基础。只要后续工作继续围绕既定主线推进，并及时将工程成果转化为规范化的论文内容，则按时完成毕业设计、中后期答辩和论文提交具有较高可行性。
